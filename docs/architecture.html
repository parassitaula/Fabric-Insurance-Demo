Technology Stack

All components are native to the Microsoft Fabric ecosystem.

🗃

Fabric Lakehouse

Unified storage layer combining the best of data lakes and data warehouses. Stores Bronze, Silver, and Gold Delta tables.

⚡

Apache Spark (PySpark)

Distributed compute engine for all data transformations. Notebooks run on Fabric Spark pools with auto-scaling.

Δ

Delta Lake

ACID-compliant storage format enabling time travel, schema enforcement, and efficient upserts across all layers.

📈

Power BI (DirectLake)

Real-time analytics directly from Delta Lake. No data movement needed — DirectLake mode reads parquet files natively.

🛠

Fabric Data Pipelines

Orchestration layer for scheduling and sequencing notebook execution across Bronze → Silver → Gold layers.

🗂

Fabric Warehouse (T-SQL)

SQL analytics endpoint for ad-hoc querying. Enables BI analysts to query Gold tables using familiar T-SQL syntax.

Data Quality Rules

Applied consistently across all Silver notebooks to ensure data integrity.

⚠ Null Validation — Required fields are checked for null and empty-string values. Rows failing validation are routed to quarantine tables with a _rejection_reason column.
🔎 Deduplication — Each Silver table is deduplicated on its business key (e.g., customer_id, policy_id). When duplicates exist, the row with the latest _ingested_at timestamp is kept.
🔧 Type Casting — String columns from CSV are cast to appropriate types: dates to DateType, monetary amounts to DoubleType. Invalid casts result in null values caught by validation.
📄 String Trimming — All string columns are trimmed of leading/trailing whitespace before validation to prevent false negatives.
🕒 Metadata Lineage — Bronze adds _source_file and _ingested_at. Silver and Gold add _processed_at. Enables full data lineage tracking.
✅ Idempotent Processing — All notebooks use overwrite mode with overwriteSchema: true. Re-running any notebook produces identical results without side effects.
📊 Quarantine Reporting — Each Silver notebook prints counts of valid vs. rejected rows. Quarantine tables are available for inspection and remediation.

Naming Conventions

Standardized naming patterns from the project constitution ensure consistency across all artifacts.

lh_<layer>_<domain> Lakehouse — e.g., lh_bronze_insurance, lh_silver_insurance, lh_gold_insurance
nb_<##>_<layer>_<name> Notebook — e.g., nb_01_bronze_ingest, nb_09_gold_claims_summary
pl_<layer>_<domain> Pipeline — e.g., pl_bronze_insurance, pl_silver_insurance
wh_<domain> Warehouse — e.g., wh_insurance
sm_<domain> Semantic Model — e.g., sm_insurance
rpt_<domain>_<purpose> Report — e.g., rpt_insurance_claims_dashboard
<table>_quarantine Quarantine table — e.g., customers_quarantine, claims_quarantine
